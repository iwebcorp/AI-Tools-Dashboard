# AI Usage Dashboard

## 1. 프로젝트 기본 정보
- 프로젝트명: AI Usage Dashboard
- 서비스 성격: OpenAI, Gemini, Cursor, ChatGPT Web, Figma 사용량과 비용을 한곳에서 확인하는 대시보드
- 공개 여부: 사내 / 개인용 대시보드
- 기간: 진행 중

## 2. 프로젝트 배경
- 여러 AI 도구의 사용량과 비용이 서비스별로 분산되어 있어 전체 현황을 한 번에 보기 어려웠습니다.
- 세션 정보와 사용량 데이터를 같은 형식으로 모아 조회할 수 있는 구조가 필요했습니다.
- 운영 중에는 연결 상태, 사용량 추이, 비용 변화를 빠르게 확인할 수 있어야 했습니다.

## 3. 내 역할
- Next.js 기반 대시보드와 API 구조를 설계 및 구현했습니다.
- 서비스별 사용량 응답을 공통 형식으로 정리했습니다.
- Chrome Extension에서 가져온 세션 정보를 Redis에 저장하고, API에서 이를 우선 조회하는 흐름을 구성했습니다.
- 사용량 차트, 서비스 카드, 상세 탭 등 핵심 화면을 구현했습니다.

## 4. 기술 구성
- Frontend: Next.js 16, React 19, TypeScript, Tailwind CSS
- Backend: Next.js API Route
- Infra / Deployment: Vercel, Upstash Redis
- 외부 연동 / API: OpenAI, Gemini, Cursor, ChatGPT Web, Figma
- 시각화: Recharts
- 검증 / 스키마: Zod

## 5. 핵심 작업
- 여러 AI 서비스의 사용량을 하나의 화면에서 볼 수 있도록 통합 대시보드를 구성했습니다.
- 서비스별 API 응답을 공통 타입으로 정리하고, 전체 조회와 서비스별 상세 조회를 분리했습니다.
- 기간 필터를 적용해 월간 및 일간 사용량을 조회할 수 있도록 했습니다.
- 캐시를 적용해 반복 조회 시 응답 부담을 줄였습니다.
- Cursor와 Figma는 계정 단위 상세 정보까지 볼 수 있도록 별도 화면을 구성했습니다.

## 6. 문제 해결
- 브라우저와 서비스별로 흩어진 세션 정보를 Redis 우선 구조로 정리했습니다.
- 서비스마다 응답 구조와 에러 유형이 달라 통합 조회가 깨지지 않도록 공통 데이터 모델을 설계했습니다.
- Cursor, ChatGPT, Figma처럼 인증 방식과 데이터 구조가 다른 서비스를 같은 화면에서 비교할 수 있도록 분기 로직을 나눴습니다.
- 외부 API 실패가 전체 대시보드 중단으로 이어지지 않도록 서비스 단위로 실패를 분리했습니다.

## 7. 성능 / 품질
- 서비스별 응답에 캐시 TTL을 적용해 동일 질의 반복 시 재조회 비용을 줄였습니다.
- Zod로 요청 값과 기간 범위를 검증해 잘못된 조회 조건을 방지했습니다.
- 서비스 상태, 토큰 수, 요청 수, 비용을 공통 타입으로 관리해 화면과 API 간 불일치를 줄였습니다.
- 일간 및 월간 차트를 통해 사용량 추이를 빠르게 확인할 수 있도록 했습니다.

## 8. 배포 / 운영
- Vercel에 배포해 웹 대시보드와 API를 함께 운영했습니다.
- Upstash Redis를 세션 저장소로 사용해 확장 가능한 구조를 구성했습니다.
- Chrome Extension으로 세션 수집과 서버 동기화를 분리했습니다.
- 서비스별 연결 상태와 오류 메시지를 노출해 운영 중 원인 파악과 대응이 가능하도록 했습니다.

## 9. 검증 자료
- 공개 URL: 별도 공개 범위 내에서 확인 가능
- 메인 화면: 전체 서비스 사용량 요약과 차트
- 상세 화면: OpenAI / Gemini / Cursor / Figma / ChatGPT 개별 탭
- 보조 자료: Chrome Extension 연동 화면, 서비스별 API 응답

## 10. 결과
- 여러 AI 도구의 사용량과 비용을 한 화면에서 확인할 수 있도록 했습니다.
- 서비스별 토큰, 요청 수, 비용 추이를 비교할 수 있는 기준을 마련했습니다.
- 운영 관점에서 연결 상태와 이상 징후를 빠르게 확인할 수 있도록 했습니다.
- 분산된 데이터를 단일 대시보드로 통합해 관리 효율을 높였습니다.

## 11. 회고
- 이 프로젝트는 단순 UI 구현보다 데이터 정규화, 서비스별 예외 처리, 운영 안정성 확보가 핵심이었습니다.
- 이직용 설명에서는 화면 구현보다 분산된 외부 데이터를 통합하는 설계 경험을 강조하는 편이 적절합니다.
- 정량 지표는 실제 측정값이 있을 때만 넣는 것이 좋습니다.
